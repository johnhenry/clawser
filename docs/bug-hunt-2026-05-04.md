## #1 — `clawser config set max_tokens` was a stub (FIXED)

**Severity:** Important — the CLI command lied to the user, claiming
to apply a value that was never wired anywhere.

**Location:** `web/clawser-cli.js`, `cmdConfig` set:max_tokens branch
(originally line 350).

### What the bug was

```js
case 'max_tokens':
case 'max-tokens': {
  const n = parseInt(value, 10);
  if (isNaN(n) || n <= 0) { /* error */ }
  if (json) return jsonOut({ key: 'max_tokens', value: n }, ...);
  return { stdout: `max_tokens noted: ${n} (applied at request time)\n`, ... };
}
```

The output literally said `applied at request time` — but no code
read this value at request time. The agent had no `maxTokens` field
at all; the CLI parsed and validated, then echoed and discarded.

### What the fix does

- Added `#defaultMaxTokens` field and `setDefaultMaxTokens(n)` /
  `getDefaultMaxTokens()` methods on `ClawserAgent`.
- The two `_providerOpts` construction sites in `agent.run()` now
  populate `_providerOpts.max_tokens = this.#defaultMaxTokens` when
  set, so the value flows into every `provider.chat` /
  `provider.chatStream` call.
- `agent.persistConfig()` writes `config.maxTokens` alongside provider
  and model.
- `applyRestoredConfig` re-applies `savedConfig.maxTokens` after
  `onProviderChange` (matching the model-restoration pattern).
- The CLI command now calls `agent.setDefaultMaxTokens(n)` and
  `persistConfig()`, with a legacy-mock fallback for tests that
  don't expose those methods. Output message corrected to
  `max_tokens set to: N`.

### Tests

`web/test/clawser-cli-max-tokens.test.mjs` — 5 tests covering valid
integer / dash-aliased flag / zero / non-integer / legacy agent
fallback.

### Why this matters

A user typing `clawser config set max_tokens 4096` had every reason
to believe the limit was now in effect. Before the fix, every
subsequent request still used the provider default — silently
overrunning what the user thought their budget was. Most provider
APIs respect a server-side default cap, but the bug made the CLI's
config surface untrustworthy.

## #2 — 15+ setIntervals without `.unref()` (SURFACED)

**Severity:** Minor — does not affect production correctness; affects
test cleanup and Node-side process exit. Tests work today via the
test runner's drain-then-kill workaround, but adding `.unref()` would
let processes exit naturally without that band-aid.

**Locations** (all production code, none in test files):

```
web/clawser-routines.js:334            #cronTicker
web/clawser-mesh-sync.js:322           autoSync intervals
web/clawser-mesh-swarm.js:867          ping interval
web/clawser-presence.mjs:83            sweep timer (already unref'd ✓)
web/clawser-peer-session.js:426        heartbeat
web/clawser-workspace-lifecycle.js:842 _updateInterval
web/clawser-peer-memory-sync.js:270    syncInterval
web/clawser-mesh-websocket.js:350      heartbeat
web/clawser-mesh-hardening.js:333      ping
web/clawser-mesh-hardening.js:659      eviction
web/clawser-daemon.js:311              tab heartbeat
web/clawser-daemon.js:458              auto-checkpoint
web/clawser-daemon.js:523              auto-checkpoint (alt path)
web/clawser-channel-discord.js:196     heartbeat
web/clawser-mesh-discovery.js:733      exchange
```

### Why not fix inline

Each intervals' lifetime is tied to its owning class (mesh swarm,
discovery, daemon, etc.). Adding `.unref()` is a one-token mechanical
edit per call site, but each call site needs a code-path review to
confirm `.unref()` is actually correct semantically — i.e., the
interval doesn't represent the only thing keeping the node alive.

The failure mode of getting one wrong: process exits while a
work-loop interval is still pending → silent loss of in-flight work
(e.g. auto-checkpoint, heartbeat). That's a bigger risk than the
current "tests use a workaround" cost.

### Recommended fix

A targeted pass per-module: for each interval, decide whether the
work it does is meaningful in a backgrounded Node process. For the
test-impact cases (presence, mesh-sync auto-sync, swarm ping,
mesh-discovery exchange) → safe to unref. For the
ownership-bound ones (daemon auto-checkpoint, peer-session heartbeat
during an active session) → leave as-is and rely on explicit
`stop()`/`shutdown()` paths.

Estimated effort: S (1-2 hours, 15 careful per-call-site decisions).

## #3 — `void openRemoteRuntimeView(...)` swallows rejections (SURFACED)

**Severity:** Nitpick — fire-and-forget UI handlers in browsers
generate `unhandledrejection` events, so the worst case is a console
warning.

**Location:** `web/clawser-workspace-init-mesh.js:424`

```js
initRemoteRuntimePanelListeners({
  onOpenView: (selector, view) => {
    void openRemoteRuntimeView(selector, view);  // ← rejection unhandled
  },
  ...
});
```

If `openRemoteRuntimeView` rejects, the user sees nothing — no error
message, no toast, no log entry. Diagnosis would require opening the
DevTools console to see the `unhandledrejection`.

### Recommended fix

```js
onOpenView: (selector, view) => {
  openRemoteRuntimeView(selector, view).catch(err => {
    console.warn('[clawser] open remote view failed:', err?.message || err);
  });
},
```

Trivially safe; a one-line edit. Not done in this pass to keep the
bug-hunt diff focused on the one real bug found.
